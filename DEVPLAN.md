# PuckSim — Development Plan

Translation of [DESIGN.md](DESIGN.md) into an ordered, dispatchable sequence of build steps. Each step is scoped to roughly "one PR's worth" of work — small enough to hand to a single coding agent, big enough to be a coherent unit.

All HoopR file references below were verified directly against `/Users/hank/HoopR/hoopsim/` source (paths, line counts, and class/function signatures accurate as of the planning session on 2026-07-01).

Conventions:
- **Phase**: MVP / v1 / v2+ (per DESIGN.md phasing)
- **Deps**: step numbers that must merge first
- **HoopR ref**: real file(s) to mirror the *pattern* of (not a fork — rewrite the numbers, reuse the shape)
- **Files**: new/modified PuckSim paths (all under `/Users/hank/PuckSim/`)
- **Done**: concrete, checkable exit criteria

### Step 1.7 — team.py: roster + line/pair on-ice-group data structures
**Deps:** 1.6 (hard blocking)
**HoopR ref:** `hoopsim/models/team.py` (419 lines — `roster: List[int]` (ids only, pure data), `chemistry`/`pair_key()`/`lineup_familiarity_secs()`/`seed_chemistry()`, `roles`/`ROLE_TAGS`, `tactics`, `coach`, win/loss/streak, `dead_money`, roster-helper functions outside the dataclass: `roster_players()`, `team_salary()`, `assign_positions()`, `auto_set_lineup()`, `rotation_pool()`).
**Files:** `pucksim/models/team.py`
**Critical decision (DESIGN.md point 1):** on-ice groups must be plain lists, not hard-coded `Line`/`Pair` classes. Concretely: `Team.lines: List[List[int]]` (4 x 3 ids), `Team.pairs: List[List[int]]` (3 x 2 ids), `Team.goalie_starter`/`Team.goalie_backup: Optional[int]`. Expose `current_forward_line(idx)`/`current_d_pair(idx)` returning plain lists so a future "caught players" pass can splice ids in without a schema change. Reuse HoopR's `pair_key()`/`lineup_familiarity_secs()` directly over whichever 2-5 ids are grouped for a shift. Auto-line-builder mirrors `assign_positions()`'s slot-fill-by-fit algorithm for LW/C/RW/D slots.
**First sub-task:** confirm `POSITIONS` shape from Step 1.4 before writing the line-builder.

**Position flexibility & handedness (decided 2026-07-01, post-Wave-2 design amendment):** Real rosters don't hard-lock players to one forward slot — a player's `position` (LW/C/RW/D/G, set in Step 1.6) is their *primary* slot, but forwards can fill any of the 3 forward slots and D can be paired on either side, each with a fit penalty scored by the line-builder (never baked into `attributes.overall()` — that stays position-agnostic per the design discussion). Two distinct penalty axes, both **provisional/tunable, first-pass values**:
- **Position-category penalty** (primary position vs. assigned slot, forwards only — D has one slot so no category penalty applies): wing↔wing (small — e.g. RW slotted at LW), C→wing (smaller — center is "less defensively intensive" to move off of), wing→C (larger — center carries more defensive/faceoff responsibility). Add as `POSITION_FIT_PENALTY: Dict[Tuple[str,str], int]` constants in `config.py` (extending the already-committed file), consumed by team.py's line-builder fit-score, not by `overall()`.
- **Handedness/side penalty** (new `Player.shoots: "L"|"R"` field, added directly to `player.py` in this same design pass): applies independently of position — a natural side mismatch (e.g. a right-shot on left wing, or a same-handed D pair instead of a lefty/righty pair) costs a smaller fit penalty than a position-category mismatch. D pairing in the auto-line-builder should *prefer* opposite-handed pairs (real-NHL norm) but same-handed pairs remain legal, just penalized. Add matching handedness-penalty constants alongside `POSITION_FIT_PENALTY` in `config.py`.
- D stays a single `POSITIONS` slot (no LD/RD split) per this design pass — explicitly flagged to revisit later if the single-slot model turns out to lose too much real fidelity, once actual line-building/gameplay data exists.
**Done:** `tests/test_team.py` — auto-line-builder on a 12F/6D/2G roster produces 4 complete lines + 3 complete pairs + starter/backup goalie; `Team.lines`/`Team.pairs` are plain `List[List[int]]` (test splices an extra id in with no type error, proving the flexibility requirement); fit-score is strictly lower for an off-position slotting than an on-position one, and lower still for wing→C than C→wing or wing↔wing (position-category penalty ordering); a same-handed D pair scores lower than an opposite-handed D pair, all else equal (handedness penalty).

## PHASE 2 — v1 (special teams, goalies, faceoffs/injuries, cap/trades/FA/draft, playoffs+OT/SO, awards, web app)

Larger units than MVP — each still independently assignable, sized like "one subsystem."

**Scope note (2026-07-01):** this dispatch round covers gameplay systems only (Steps 2.1–2.8). Steps 2.9/2.10 (FastAPI backend + React frontend) are planned as a separate round once these land, per DEVPLAN's own guidance that the web layer should wait until "enough systems exist to expose."

**MVP technical debt these steps must resolve:** the MVP engine shipped two explicitly-provisional placeholders that v1 replaces for real:
- `sim/engine.py`'s `coach_session()` OT block: "one extra simplified 5v5 sudden-death period... real 3-on-3/shootout resolution is Step 2.6" — literally replace this code block in Step 2.6.
- `sim/season.py`'s `_apply_result()`: a skill-weighted coin-flip placeholder tiebreak for `has_shootout=True` rules when the engine returns an unresolved tie. Once Step 2.6 makes the engine itself always resolve `has_shootout=True` games decisively (via real shootout simulation), this season.py placeholder becomes dead code for that path — Step 2.6's owner should remove it (or leave it as a documented defensive fallback if some edge case can still produce an unresolved tie — judgment call, but don't leave two competing tie-resolution mechanisms silently coexisting).

**Coach knobs already sitting unused, waiting for these steps to consume them** (all defined on `CoachProfile` in `pucksim/models/coach.py`, per that file's own docstring: "consumed starting at Step 1.12/2.1/2.2/2.8"): `pp_forwards` (2.1), `pk_style_aggression` (2.1), `defensive_risk_tolerance` (2.1, D pinching/gap control), `goalie_pull_max_deficit`/`goalie_pull_time_threshold_secs` (2.2), `line_juggling_patience` (2.8 — now wired to a real reshuffle trigger, see that section).

**Known low-priority gap, found independently by both Step 2.7 and Step 2.8 (2026-07-02), not fixed:** `models/coach.py`'s `Coach.to_dict()` only serializes the archetype NAME, not a full profile — a custom `CoachProfile` not registered in `ARCHETYPES` silently resolves back to `Balanced` on `from_dict()` (`profile_for()`'s documented never-crash fallback). **Update (Step 2.7 review, 2026-07-02): same root cause confirmed in a second test, only partially mitigated, still not fixed.** Step 2.7 narrowed both `test_corsi_and_fenwick_tallied_as_event_stream_filter` and a new `test_goal_updates_plus_minus_nets_to_zero_when_every_goal_is_5v5` to search for a "clean" scenario (a game where every relevant event reports `strength_state == STRENGTH_5V5` and no empty-net goal occurred) before asserting — a real improvement (failure rate dropped from ~5% of seeds to under 1%), but NOT airtight, and independently verified as still reachable: a team pulling its OWN goalie for a 6th attacker never changes `strength_state` (goalie-pull is orthogonal to the penalty-driven strength machine, confirmed by direct trace) and produces no distinguishing PBP signal at all when that team's own shots are the ones in question (the shot lands on the *opponent's* still-staffed net, so `goalie_id` is never `None` on it) — so a "5v5-labeled, no-empty-net-goal-anywhere" game can still legitimately contain a 6-on-5 shift and break the exact-reconciliation invariant. The only complete fix remains the same as before: a per-event on-ice-group-size field on `PBPEvent` (or equivalent live-state access the test can assert against directly, bypassing `GameResult`'s summary-only view). Still not urgent (both tests pass today, the residual failure rate is low, and this is well-trodden ground for whichever step finally touches `PBPEvent`'s schema) but now affects two tests instead of one.

**Three-way faceoff resolution (design input, 2026-07-02):** MVP's `_log_faceoff` is a plain two-way coin flip between the two centers' `faceoffs` rating. Real draws are often genuinely tied up (a clean win is not the only outcome) — this step should resolve faceoffs as a three-way roll (home center clean win / away center clean win / **tie**, a contested/scrum draw) rather than a binary one. This three-way split is purely a *simulation-accuracy* mechanic, not a new player-facing stat category: `fo_won`/`fo_lost` stay exactly the binary box-score fields they are today (real NHL box scores have no "tie" faceoff stat — a draw always eventually goes to one side), so the tie branch must still resolve to a single final winner/loser pair before touching the box score, it just gets there through an extra step. Whichever team wins that secondary roll is credited as the faceoff winner in the box score exactly as if their center had won cleanly (the tie/scrum is an internal simulation detail, invisible to `fo_won`/`fo_lost`) — but this step should still log which path produced the result (clean win vs. won-off-a-tie) in the `EVENT_FACEOFF` PBP event's context, since that's useful signal for a future zone-entry/possession-quality model even though it's not a tracked stat today. This finally gives faceoff outcome real teeth for the icing/offside-stoppage possession-gating this step is already adding (see this section's intro) — a scrum-then-winger-recovery draw is a believably *worse* start to a shift than a clean win, which is exactly the kind of nuance worth capturing now that faceoffs start mattering to zone/possession instead of being pure box-score flavor.
**Done:** `tests/test_faceoffs.py` — win probability monotonic in center rating gap, icing/offside faceoffs correctly logged with the right stoppage-type context; the three-way tie/winger-recovery path is exercised directly (a forced-tie scenario resolves via the winger roll, is still binary in `fo_won`/`fo_lost`, and is distinguishable in PBP context from a clean win). `tests/test_injuries.py` — injury rates within a sane band over a season; injured players excluded from lineup/rotation selection until healed; full `tests/test_engine.py`/`test_season.py` still pass.

**Deps:** 2.1–2.3 (real stat categories — hits, blocks, goalie starts, penalty minutes — to compute awards/development over; technically only needs 1.13/1.6 per data shape, but awards over 5v5-only MVP stats would be thin)
**HoopR ref:** `hoopsim/systems/awards.py` (108 lines — `compute_awards()`, swap MVP/DPOY/ROY/MIP/All-League for Hart/Norris/Vezina/Calder/Selke); `hoopsim/systems/momentum.py` (111 lines — `update_morale()`/`offseason_reset()`/`game_score()`, generic-framework carryover); `hoopsim/systems/legacy.py` (186 lines — HOF resume-snapshot pattern); `hoopsim/systems/development.py` (73 lines — age curve, `_overall_delta()`); `hoopsim/systems/offseason.py` (178 lines — `archive_season()` orchestration order: awards → career archival → development).

**Goalie year-to-year consistency (design input, 2026-07-01 — a `development.py` requirement, not a Step 2.2 in-game concern):** real NHL goalies are notably inconsistent year-to-year compared to skaters — only a small elite tier (~top 3-5 in the league; contemporary examples the user cited: Vasilevskiy/Shesterkin/Hellebuyck-type goalies, not always Vezina finalists but reliably good-to-very-good every year) stays tightly banded near their true talent level season after season. Below that tier is a "squishy middle" group where a given goalie might post a breakout year, a down year, or a flatly average year almost unpredictably. `attributes.py`'s `gk_consistency` rating already exists (part of `GOALIE_WEIGHTS`, currently just a 0.10-weighted skill component feeding `overall()`) — this step should repurpose/extend it to ALSO drive season-to-season output *variance*, not just skill level: at each offseason transition, resample a per-season "form" modifier for each goalie with a spread inversely proportional to their `gk_consistency` (high consistency = tight spread, stays near their established rating; low consistency = wide spread, can swing to a breakout or bust season). Keep this distinct from the permanent age-based development curve (`_overall_delta()`) — this is a temporary, resampled-every-season modifier layered on top, not a permanent rating change; a bust or breakout year shouldn't permanently drag a goalie's true `ratings` dict up/down the way real aging does.
**Important distinction from this session's "no upweighting" principle** (see the `[[feedback_no_upweighting]]` memory and this phase's intro note): that principle governs deterministic *in-game* realization mechanics (morale/clutch/hot-hand within a single game), which must never let a player exceed their rating ceiling on any given shot/shift. Season-level "form" variance is a different, legitimate kind of randomness — natural statistical scatter around a goalie's unchanging true talent, symmetric (can land above or below their established level in a given season), with `gk_consistency` controlling the width of that scatter, not a one-directional mechanic. Don't conflate the two when implementing this — document the distinction clearly in `development.py` so a future reader doesn't "fix" this into a no-upweighting violation by mistake.

**Generation-time rarity correlation (design input, 2026-07-01, companion to the above — touches `gen/playergen.py`, not just `development.py`):** high skill *and* high `gk_consistency` together should be RARE — that combination is what makes a true "franchise goalie" (Vasilevskiy/Shesterkin/Hellebuyck-caliber) special and scarce in real hockey. A bad goalie being consistently bad is common and unremarkable; a good goalie being consistently good is the rare case worth gating. `gk_consistency` must NOT be sampled independently/uniformly of skill in `generate_goalie()` — it needs a skill-correlated rarity gate. Concrete mechanism: when a generated goalie's calibrated overall lands at or above a high-skill threshold (e.g. roughly the top 15-20% of goalie talent — exact cutoff is a tuning choice), sample `gk_consistency` from a capped low/mid band by default (representing the much more common "talented but streaky" case) UNLESS a separate, small-probability "reliability roll" succeeds (e.g. 5-10%, tunable), in which case `gk_consistency` is instead sampled from the elite band — this is what makes the elite-skill-plus-elite-consistency combination genuinely rare (it requires winning both an independent skill roll AND a low-probability reliability roll). Below the high-skill threshold, sample `gk_consistency` normally/independently with no gating — an average or backup-level goalie can freely land anywhere on the consistency spectrum, matching "bad goalies can be consistently bad" being unremarkable. This pairs naturally with the existing `RARE_GOALIE_ARCHETYPES` tier (currently just "Generational Goalie") but is a distinct, additional gate specifically on `gk_consistency` vs. overall skill, not the same mechanism as rare-archetype assignment.
**Files:** `pucksim/systems/{awards,legacy,momentum,offseason,development}.py`; also revisits `pucksim/gen/playergen.py`'s `generate_goalie()` for the rarity-correlation gate above (a targeted amendment to existing MVP code, not a rewrite).
**Done:** `tests/test_awards.py` — award eligibility gating sensible (Hart/Norris/Vezina/Calder/Selke); `tests/test_momentum.py`/`test_offseason.py`/`test_legacy.py` mirror HoopR's test shapes; `tests/test_development.py` — over many simulated offseasons, high-`gk_consistency` goalies show a measurably tighter season-to-season output distribution than low-`gk_consistency` goalies, and the average across many seasons for any given goalie stays centered near their true rating (proving the variance is symmetric scatter, not a systematic drift); `tests/test_generation.py` (extends existing) — across a large generated goalie pool, the fraction landing in BOTH the high-skill AND high-`gk_consistency` bands is measurably small relative to high-skill-alone or high-consistency-alone, confirming the rarity gate actually creates scarcity rather than independence.

**Open item, carried forward (Step 2.7 review, 2026-07-02): the goalie season-form mechanic is built but not yet CONSUMED by gameplay.** `systems/development.py`'s `GoalieFormState`/`resample_goalie_form`/`apply_goalie_form` are fully implemented, tested (statistically verified: a `gk_consistency=99` goalie's form stdev is ~4.2x tighter than a `gk_consistency=25` goalie's, both means centered on `FORM_BASELINE=1.0`), and wired into `systems/offseason.py`'s per-World state (mirroring `sim/goalies.py`'s `GoalieRestState` id()-with-reference pattern). But `apply_goalie_form` is never actually called from `sim/engine.py`/`sim/goalies.py`'s in-game save-probability math — this step was explicitly scoped to stay out of those files (Step 2.2's territory), so the mechanic currently has ZERO observable effect on simulated games; it's data-layer infrastructure only, the same kind of "stub awaiting a future wiring step" as `Team.tactics` before Step 2.8. Whichever future step next opens up `sim/goalies.py`'s save-probability formula should fold `offseason.goalie_form_state(world)` + `development.apply_goalie_form(...)` into the goalie's effective rating for that season (already flagged inline in `offseason.py`'s `goalie_form_state()` docstring as "a future save-probability integration in sim/goalies.py").
